## 二、功能设计与玩法机制

### 2.1 核心抽盒流程

```
选盒 → 摇盒(获线索) → 拆盒(出结果) → 确认/分享
```

**各环节细节：**

| 环节 | 玩法说明 |
|------|---------|
| **选盒** | 展示系列款式图，提示各款概率分布 |
| **摇盒** | 线上模拟"摇一摇"获取线索（重量、颜色、特点描述）；可消耗积分或付费获得提示卡 |
| **拆盒** | 播放开盒动画，记录到"我的盲盒"库存 |
| **确认** | 提供"再抽一次"或"分享开盒"选项；部分平台支持"一键回盒"（未拆封置换） |

### 2.2 主流玩法模式

| 玩法 | 描述 | 典型案例 |
|------|------|---------|
| **直接购买** | 按定价随机抽取 | 泡泡玛特、名创优品 |
| **十连抽** | 一次性抽10次，通常有折扣/保底 | 几乎所有小程序 |
| **抽签** | 限量款先报名抽资格，中签后付款 | 潮玩展会、联名限量 |
| **合成** | 收集指定款式合成更稀有的款式 | 《原神》周边、数字藏品 |
| **兑换** | 重复款/积分兑换指定款 | 会员商城 |
| **集卡** | 收集整套款式激活额外奖励 | 泡泡玛特"集齐换隐藏" |
| **摇盒/提示卡** | 付费获取款式线索 | 泡泡玛特抽盒机 |
| **拼团/助力** | 邀请好友获得额外抽盒次数 | 社交裂变玩法 |

### 2.3 社交功能

- **交换市场**：用户发布"我有→想要"，匹配后线下/平台内交换
- **晒图分享**：开盒后生成精美卡片，一键分享至微信/小红书/朋友圈
- **社区**：圈子话题、开盒打卡、关注互动
- **排行榜**：开盒数榜、收藏榜、积分榜，前列有奖励

### 2.4 商业模式设计

| 模式 | 设计 | 案例 |
|------|------|------|
| **会员分级** | 普通/银卡/金卡/钻石，权益递减 | 泡泡玛特会员体系 |
| **月卡** | 每月固定费用获得N次抽盒机会+积分 | 潮玩族等 |
| **订阅** | 每月自动寄送随机福袋 | 日本付费扭蛋订阅 |

## 四、概率模型与算法设计（核心）

### 4.1 概率模型方案对比

| 模型 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **固定概率** | 每次独立随机，概率恒定 | 简单、透明、易合规 | 极端情况下可能一直不出 | 普通款抽奖 |
| **硬保底** | 抽满N次必出稀有 | 体验好、可预期 | 期望概率抬升，成本高 | 隐藏款保底 |
| **软保底** | 概率随次数递增 | 平滑、保持刺激感 | 实现复杂、难解释 | 主流游戏（原神等） |
| **混合模型** | 固定概率+软/硬保底 | 兼顾惊喜感和公平性 | 数学模型复杂 | **推荐方案** |
| **库存池** | 预先分配库存，抽走即减 | 精确控制出货量 | 大并发下有超卖风险 | 限量发售 |

### 4.2 推荐方案：混合概率+软保底

#### 4.2.1 核心公式

```
P_base = 隐藏款基准概率（如 1%）

P_actual(n) = P_base × (1 + α × n)
  其中 n = 连续未中次数
  当 P_actual(n) ≥ 1.0 时，强制出隐藏（硬保底）
```

#### 4.2.2 参数设计

```
系列：12个普通款 + 2个隐藏款

隐藏款：基准概率 1%，软保底 60 抽，硬保底 90 抽
稀有款：概率 15%
普通款：概率 84%（均分到12款，各7%）

概率递增因子 α = 0.015（每抽递增1.5%）
期望隐藏出货 ≈ 39-42 抽（含保底机制）
```

#### 4.2.3 伪代码实现

```go
// 加权随机选择 + 软保底
type ProbItem struct {
    ID       string
    BaseProb float64 // 基准概率
    Weight   float64 // 动态权重
}

// 软保底概率计算
func calcSoftPity(baseProb float64, consecutiveMisses int, pityFactor float64) float64 {
    increasedProb := baseProb * (1 + pityFactor*float64(consecutiveMisses))
    // 硬保底：概率达到100%时必出
    if increasedProb >= 1.0 {
        return 1.0
    }
    return increasedProb
}

// 加权随机选择（Alias Method 优化版）
func weightedRandom(items []ProbItem) string {
    totalWeight := 0.0
    for _, item := range items {
        totalWeight += item.Weight
    }
    
    r := rand.Float64() * totalWeight
    cumulative := 0.0
    for _, item := range items {
        cumulative += item.Weight
        if r < cumulative {
            return item.ID
        }
    }
    return items[len(items)-1].ID
}

// 主抽奖函数
func lottery(user *User, series *Series) *Prize {
    consecutiveMisses := user.GetConsecutiveMisses(series.ID)
    
    // Step 1: 先判断是否出隐藏/稀有
    hiddenProb := calcSoftPity(series.HiddenBaseProb, 
        consecutiveMisses.SoftPity, series.PityFactor)
    
    if rand.Float64() < hiddenProb {
        // 出隐藏款
        user.ResetConsecutiveMisses(series.ID)
        return series.GetRandomHidden()
    }
    
    // Step 2: 从稀有+普通款中按权重抽取
    items := series.GetCommonAndRareItems()
    selected := weightedRandom(items)
    
    if items.IsRare(selected) {
        user.ResetConsecutiveMisses(series.ID)
    } else {
        user.IncrementConsecutiveMisses(series.ID)
    }
    
    return selected
}
```

### 4.3 库存分配算法（限量发售）

```go
// Redis + Lua 保证原子性
// 库存扣减 Lua 脚本
const subtractStockScript = `
    local key = KEYS[1]          -- 库存 key
    local user_key = KEYS[2]     -- 用户已购 key
    local user_id = ARGV[1]      -- 用户ID
    local limit = tonumber(ARGV[2]) -- 限购数量
    
    local current = redis.call("GET", key)
    if not current or tonumber(current) <= 0 then
        return -1  -- 库存不足
    end
    
    local user_count = redis.call("GET", user_key .. ":" .. user_id)
    if user_count and tonumber(user_count) >= limit then
        return -2  -- 超过限购
    end
    
    redis.call("DECR", key)
    redis.call("INCR", user_key .. ":" .. user_id)
    return tonumber(current) - 1  -- 剩余库存
`
```

### 4.4 出货率控制（库存池模型）

适用于限定款/限量发售场景：

```
// 确定开盒顺序（骰子提前生成）
box_sequence = shuffle(all_items) // 提前打乱

// 用户开盒时按顺序分配
user_gets = box_sequence[user_index]
```

**特点：**
- 精确控制每箱出货量，保证整箱出隐藏
- 但缺乏"开盒惊喜感"——用户开盒结果其实是预定的
- **合规风险**：需要告知用户是否为"预置"模式

### 4.5 概率验证方法

| 方法 | 说明 |
|------|------|
| **蒙特卡洛模拟** | 模拟100万+次抽奖，验证输出概率≈公示概率 |
| **卡方检验** | 检验实际分布与理论分布是否一致 |
| **K-S检验** | 检验保底机制下概率分布是否合理 |
| **日志审计** | 每抽出日志（用户、时间、奖池、概率），可供审计 |

## 六、建议与最佳实践

### 6.1 功能优先级建议（MVP → V2 → V3）

**MVP（核心功能）：**
- 注册/登录 + 充值
- 盲盒展示（系列、款式、概率公示）
- 核心抽盒（单抽 + 十连抽）
- 库存管理（我的盲盒）
- 概率系统（混合模型+软保底）

**V2（增长功能）：**
- 积分/会员体系
- 摇盒线索/提示卡
- 交换市场
- 社交分享

**V3（生态功能）**
- 合成/兑换系统
- 社区/排行榜
- 月卡/订阅
- 直播开盒

### 6.2 技术选型建议

| 模块 | 推荐方案 |
|------|---------|
| **随机数生成** | `crypto/rand`（Go）或 `secrets`（Python）—— 密码学级别PRNG |
| **加权随机** | Alias Method（O(1)时间，适合大规模抽奖） |
| **库存扣减** | Redis + Lua 脚本（保证原子性） |
| **概率配置** | 数据库/配置中心（动态调整，无需发版） |
| **审计日志** | 每个抽奖事件记录：user_id, prize_id, timestamp, probability |
| **防刷策略** | IP限流、用户频控、设备指纹 |

### 6.3 合规红线

1. ✅ **必须公示所有款式的抽中概率**
2. ✅ **不得向8岁以下销售，8-16岁需监护人同意**
3. ✅ **单价不超过200元**
4. ❌ **不得将盲盒设计成"赌博"形式（如现金盲盒、提现盲盒）**
5. ✅ **保底机制需写在产品说明中**
6. ✅ **保留抽奖日志至少1-2年备查**
