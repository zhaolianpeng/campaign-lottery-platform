## 1. 产品定位与目标用户

### 1.1 产品定位

**「BOX·MAGIC」** 是一款面向年轻消费者的**线上盲盒抽奖平台**。用户通过购买盲盒获得随机礼品，集齐完整系列可解锁隐藏奖励，并可在平台内进行盲盒交换。平台融合了**抽奖的刺激感、收藏的成就感、社交的互动性**三大核心体验。

### 1.2 核心价值主张

| 维度 | 描述 |
|------|------|
| 好玩 | 开盒动画、概率递增、摇盒提示，还原线下盲盒体验 |
| 公平 | 概率全透明公示，软/硬保底机制保障体验 |
| 社交 | 交换市场、集卡排行、分享裂变，形成用户社交网络 |
| 轻量 | H5/小程序形态，即开即用，无需下载 |

### 1.3 目标用户

| 用户画像 | 特征 | 需求 |
|----------|------|------|
| **Z世代年轻人（18-28岁）** | 追求新鲜感、热爱收藏、活跃于社交网络 | 低门槛娱乐、收集成就感、社交分享 |
| **盲盒爱好者** | 有线下抽盒/潮玩收藏经验 | 线上便捷抽盒、交换补全系列 |
| **泛娱乐用户** | 对抽奖/开盲盒有好奇心 | 碎片化时间的轻娱乐、小确幸体验 |

### 1.4 用户场景

1. **小明（大学生）**：睡前刷手机→看到朋友分享的开盒动画→点进小程序→花9.9抽了一个「星空系列」盲盒→抽到稀有款→截图分享朋友圈
2. **小丽（白领）**：收集「猫咪系列」缺最后一只→在交换市场用重复款匹配→成功换到缺的款式→集齐整套→解锁隐藏奖励
3. **阿豪（盲盒收藏家）**：查看概率公示→计算保底策略→十连抽冲刺隐藏款→中奖后发布开盒炫耀视频

## 3. 用户端功能

| 排序筛选 | 按热度、上新时间、价格排序；按状态（全部/进行中/已售罄）筛选 |
| 进度预览 | 每个系列展示"已集齐 xx/xx 款"的进度条(对已抽过的用户) |
| 下拉刷新 | 分页加载，每次返回 20 条 |

#### 3.1.2 系列详情页

| 功能 | 描述 |
|------|------|
| 系列信息 | 封面轮播图、系列名称、简介、价格、总款式数 |
| 款式一览 | 网格展示该系列所有款式（已拥有的显示"✓已拥有"角标） |
| 我的进度 | 已拥有 x 款 / 共 x 款，进度条，集齐后显示"已集齐"勋章 |
| 概率公示 | 入口按钮→弹出概率公示弹窗 |
| 操作区 | [单抽] [十连抽] 按钮；价格显示；当前积分余额 |

#### 3.1.3 概率公示页

| 功能 | 描述 |
|------|------|
| 分级概率 | 普通款 xx% / 稀有款 xx% / 隐藏款 xx% / 限定款 xx% |
| 保底说明 | "每抽 x 次未出稀有，则下次概率提升" 等文字描述 |
| 公示规则 | 包含抽奖规则说明、礼品发放说明、免责声明 |
| 合规标注 | "概率信息最后更新：2026-xx-xx"；含文化部备案号（如有） |

### 3.2 抽盒流程

#### 3.2.1 单抽流程

```
用户点击[单抽]
    │
    ▼
扣除积分/支付金额
    │
    ▼
调用抽奖引擎 (lottery engine)
    │
    ▼
系统抽奖 → 确定礼品
    │
    ├─ 普通款 → 直接揭示结果
    │
    ├─ 稀有款 → 摇盒动画 → "哇，金光！" → 揭示
    │
    └─ 隐藏款 → 特殊动画 → "超级稀有！！" → 揭示
    │
    ▼
开盒动画播放 (≈2秒)
    │
    ▼
结果页面：
  - 礼品展示（图片+名称+稀有度标签）
  - 是否重复款标注
  - [分享] [再来一盒] [查看库存] 按钮
```

#### 3.2.2 十连抽流程

```
用户点击[十连抽]
    │
    ▼
扣除积分/支付金额（享受 9.5 折优惠或折扣定价）
    │
    ▼
调用抽奖引擎 10 次（批量模式，优化性能）
    │
    ▼
开盒动画：依次展示 10 个结果
    │
    ▼
汇总页：
  - 本次获得礼品列表（按稀有度排序）
  - 新增款式标注 "NEW!"
  - 重复款标注 "x2"
  - [一键分享] [再来十连] [查看库存]
```

#### 3.2.3 摇盒提示（预热体验）

| 功能 | 描述 |
|------|------|
| 触发时机 | 用户进入系列详情页后，未抽盒前或频繁点击单抽按钮时 |
| 提示内容 | "据大数据分析，本系列当前隐藏款热度较高" / "已有 xx 位用户抽到此款" |
| 交互方式 | 手机震动感 + 轻提示文案，增加期待感（非必现，随机触发） |
| 设计目的 | 模仿线下摇盒体验，增加用户停留时长和转化率 |

### 3.3 我的盲盒

#### 3.3.1 库存管理

| 功能 | 描述 |
|------|------|
| 系列归类 | 按系列分组展示所有已获得的礼品 |
| 款式视图 | 网格展示，每个款式显示：名称、图片、稀有度标签、拥有数量 |
| 排序 | 按系列、按稀有度、按获得时间排序 |
| 搜索 | 按款式名称搜索 |

#### 3.3.2 重复款标识

- 数量 > 1 的款式右上角显示 "×N" 角标
- 列表页顶部快捷筛选「只看重复款」，方便用户快速进入交换市场

#### 3.3.3 集齐进度

- 每个系列独立显示进度：`已收集 x / 共 y 款`
- 进度达到 100% 时触发**集齐特效**（勋章动画）
- 集齐后显示解锁的隐藏奖励信息

### 3.4 盲盒交换市场

#### 3.4.1 核心匹配逻辑

```
用户A:  我想要 [款式X]  →  用户B: 我有 [款式X] 且想要 [款式A]
        我有多余 [款式A]  ←  用户B: 我多余 [款式X]

匹配成功条件：
  1. A 有 B 想要的，B 有 A 想要的（双向匹配）
  2. 双方均为重复款（拥有数量 ≥ 2 或专门选择用于交换）
  3. 双方稀有度对等（稀有↔稀有，普通↔普通，或平台补贴差价）
```

#### 3.4.2 功能列表

| 功能模块 | 描述 |
|----------|------|
| 发布交换需求 | 选择「我要」的款式 + 「我有」的款式（从库存中选择重复款） |
| 匹配推荐 | 系统自动匹配双向需求，按匹配度排序推送，支持通知提醒 |
| 交换广场 | 浏览所有用户发布的交换需求（匿名展示），可按系列/稀有度筛选 |
| 交换确认流程 | 发起交换→对方确认→系统锁定礼品→双方确认→交换完成→库存更新 |
| 交换记录 | 历史交换记录查询（成功/失败/进行中） |

#### 3.4.3 交换规则

| 规则 | 说明 |
|------|------|
| 同系列交换优先 | 相同系列内交换的手续费更低（5% vs 10% 积分消耗） |
| 跨系列交换 | 允许跨系列，但需支付更高手续费 |
| 交换冷静期 | 成功交换后 24 小时内不可对同一款式再次发起交换（防刷） |
| 稀有度映射 | 系统维护稀有度价值映射表，保障交换公平性 |

### 3.5 集卡系统

#### 3.5.1 收集机制

| 元素 | 描述 |
|------|------|
| 系列集齐 | 抽到该系列所有普通款式 + 稀有款式（隐藏款/限定款可选，由系列规则定义） |
| 集齐奖励 | 解锁专属隐藏款、限定称号、积分奖励等 |
| 进度展示 | 个人主页展示所有系列的收集进度，支持设置"收集目标" |
| 排行榜 | 按集齐系列数、限量款数量排名（周榜/总榜） |

#### 3.5.2 隐藏解锁规则

| 解锁方式 | 描述 |
|----------|------|
| 集齐解锁 | 集齐某系列全部常规款式后，自动获得该系列的隐藏款 |
| 概率抽取 | 部分隐藏款仅在抽盒中按极低概率产出（0.1%~1%） |
| 限定活动 | 节日/联名限定款通过活动任务解锁 |

### 3.6 积分/会员体系

#### 3.6.1 积分系统

| 行为 | 积分获取 |
|------|----------|
| 每日签到 | +5 积分（连续签到第 7 天 +20） |
| 单抽一次 | 消耗 100 积分（约 10 元） |
| 十连抽 | 消耗 950 积分（≈9.5 折） |
| 分享开盒结果 | +2 积分/次（每日上限 10 次） |
| 邀请新用户 | +50 积分/人（被邀请人首次抽盒再 +20） |
| 交换成功 | 消耗手续费（积分） |

#### 3.6.2 会员等级

| 等级 | 条件 | 权益 |
|------|------|------|
| Lv1 青铜 | 注册即得 | 基础抽盒权限 |
| Lv2 白银 | 累计消费 ≥ 500 积分 | 概率 +5% 上浮（不影响公示） |
| Lv3 黄金 | 累计消费 ≥ 2000 积分 | 九折券 x1/月、交换手续费 8 折 |
| Lv4 铂金 | 累计消费 ≥ 5000 积分 | 专属客服、隐藏款概率 +10% 上浮 |
| Lv5 钻石 | 累计消费 ≥ 15000 积分 | 限定款优先购、实体周边赠送 |

> **注意**：会员权益仅影响部分非公示概率，基础概率公示保持不变，符合合规要求。

## 4. 管理端功能

### 4.1 系列管理

#### 4.1.1 系列 CRUD

| 功能 | 描述 |
|------|------|
| 创建系列 | 系列名称、封面图（支持多图）、描述、价格、发售时间、总款式数 |
| 编辑系列 | 修改基本信息，已发售系列不允许修改价格和款式数 |
| 下架系列 | 软删除，已售出的礼品不受影响，仅前端不展示 |
| 系列列表 | 管理端表格展示：ID、名称、状态、创建时间、操作 |

#### 4.1.2 概率配置（系列级）

| 配置项 | 类型 | 说明 |
|--------|------|------|
| 普通款总概率 | float | 所有普通款概率之和，如 85% |
| 稀有款总概率 | float | 所有稀有款概率之和，如 13% |
| 隐藏款总概率 | float | 如 1.5% |
| 限定款总概率 | float | 如 0.5% |
| 各款式权重 | int[] | 同级别内各款式的权重分配（总和归一化） |
| 软保底次数 | int | 连续N次未出稀有后概率开始递增 |
| 硬保底次数 | int | 最多抽N次必出稀有/以上 |
| 隐藏硬保底 | int | 最多抽N次必出隐藏款（可选配置） |

### 4.2 礼品管理

#### 4.2.1 礼品 CRUD

| 功能 | 描述 |
|------|------|
| 创建礼品 | 名称、图片、所属系列、稀有度等级、SKU信息 |
| 库存管理 | 总库存、剩余库存、告警阈值（低于阈值发送通知） |
| 权重配置 | 同级别内的抽取权重值 |
| 上下架 | 单礼品上下架，下架后不会在抽奖中产出 |
| 批量操作 | 批量导入/导出礼品数据（Excel） |

### 5.2 混合概率模型

采用 **动态递增 + 保底触发** 的混合概率模型，在保障用户体验的同时控制平台出货成本。

#### 5.2.1 模型三要素

| 要素 | 说明 |
|------|------|
| **固定基础概率** | 每次抽奖的基准概率（见 5.1 分级范围） |
| **软保底 (Soft Pity)** | 连续 N 次未出稀有及以上级别时，每次增加稀有概率 |
| **硬保底 (Hard Pity)** | 最多 M 次必出稀有及以上级别；可选配置隐藏款硬保底 |

#### 5.2.2 概率递增算法

```
算法描述（伪代码）：

function computeProbabilities(user, series):
    // 1. 获取该系列基础配置
    base = getBaseProb(series)
    rareBase = base.Rare
    
    // 2. 获取用户当前连续未出稀有的次数
    streak = getUserStreak(user, series)
    
    // 3. 检查硬保底
    if streak >= config.HardPityGuarantee:
        // 硬保底触发：必出稀有或以上
        return forceRareOrAbove(series)
    
    // 4. 检查软保底（概率递增）
    pityLevel = max(0, streak - config.SoftPityThreshold)
    rareProb = rareBase + (pityLevel * config.SoftPityIncrement)
    
    // 5. 上限封顶
    rareProb = min(rareProb, config.MaxProbabilityCap)
    
    // 6. 调整普通款概率（稀有概率增加的部分从普通款扣除）
    commonProb = base.Common - (rareProb - rareBase)
    commonProb = max(commonProb, 0.0)
    
    // 7. 隐藏款、限定款保持固定概率（独立计算）
    hiddenProb = base.Hidden
    limitedProb = base.Limited
    
    // 8. 归一化处理
    total = commonProb + rareProb + hiddenProb + limitedProb
    return normalize(commonProb, rareProb, hiddenProb, limitedProb, total)
```

#### 5.2.3 概率变化示例

| 连续未出稀有次数 | 普通款概率 | 稀有款概率 | 隐藏款概率 | 限定款概率 |
|:---:|:---:|:---:|:---:|:---:|
| 0 | 80.0% | 15.0% | 4.0% | 1.0% |
| 10 | 80.0% | 15.0% | 4.0% | 1.0% |
| 29 | 80.0% | 15.0% | 4.0% | 1.0% |
| **30** (软保底触发) | **78.0%** | **17.0%** | 4.0% | 1.0% |
| 40 | 58.0% | 35.0% | 4.0% | 1.0% |
| **50** (到达上限) | **58.0%** | **37.0%** (封顶) | 4.0% | 1.0% |
| **60** (硬保底) | — | **100%** (必出稀有+) | — | — |

> **配置参数示例**：SoftPityThreshold=30, SoftPityIncrement=2%, HardPityGuarantee=60, MaxProbabilityCap=37%

## 附录

### A. 技术栈选型理由

| 组件 | 选择 | 理由 |
|------|------|------|
| 语言 | Go 1.24 | 高并发、编译快、部署简单 |
| HTTP | net/http (无框架) | 轻量、无依赖、适合中小项目 |
| 数据库 | MySQL | 关系型数据，事务支持完善 |
| 缓存 | Redis | 分布式锁、计数器、缓存热点数据 |
| 开发存储 | 内存 Map | 快速原型、单元测试友好 |

### B. 抽奖引擎性能指标

| 指标 | 目标值 |
|------|--------|
| 单次抽奖响应时间 | ≤ 100ms (P99) |
| 十连抽响应时间 | ≤ 300ms (P99) |
| 并发抽奖 QPS | ≥ 5000 |
| 库存扣减一致性 | 强一致 (MySQL事务 + Redis锁) |
