## 1. 产品定位与目标用户

### 1.1 产品定位

**「BOX·MAGIC」** 是一款面向年轻消费者的**线上盲盒抽奖平台**。用户通过购买盲盒获得随机礼品，集齐完整系列可解锁隐藏奖励，并可在平台内进行盲盒交换。平台融合了**抽奖的刺激感、收藏的成就感、社交的互动性、合成的目标感**四大核心体验。

> **v2.0 定位升级**：从「盲盒抽奖平台」升级为「潮玩娱乐社区」—— 融合抽奖、收藏、合成、交易、社交于一身的泛娱乐平台。

### 1.2 核心价值主张

| 维度 | 描述 |
|------|------|
| 好玩 | 开盒动画、概率递增、摇盒提示、AR开盒、多连开，还原并超越线下盲盒体验 |
| 公平 | 概率全透明公示，软/硬保底机制保障体验，大小保底防歪 |
| 社交 | 交换市场、集卡排行、分享裂变、组队开盒、送礼，形成用户社交网络 |
| 丰富 | 合成升级、碎片拼图、限定抢购、月度战令，玩法多样不收 |
| 轻量 | H5/小程序形态，即开即用，无需下载 |

### 1.3 目标用户

| 用户画像 | 特征 | 需求 |
|----------|------|------|
| **Z世代年轻人（18-28岁）** | 追求新鲜感、热爱收藏、活跃于社交网络 | 低门槛娱乐、收集成就感、社交分享 |
| **盲盒爱好者** | 有线下抽盒/潮玩收藏经验 | 线上便捷抽盒、交换补全系列、合成稀有 |
| **泛娱乐用户** | 对抽奖/开盲盒有好奇心 | 碎片化时间的轻娱乐、小确幸体验 |
| **策略型用户** | 精算概率、追求保底最优解 | 十连打折、战令通行证、保底计算器 |

### 1.4 用户场景

1. **小明（大学生）**：睡前刷手机→看到朋友分享的开盒动画→点进小程序→花9.9抽了一个「星空系列」盲盒→抽到稀有款→截图分享朋友圈
2. **小丽（白领）**：收集「猫咪系列」缺最后一只→在交换市场用重复款匹配→成功换到缺的款式→集齐整套→解锁隐藏奖励
3. **阿豪（盲盒收藏家）**：购买月卡→查看概率公示→计算保底策略→十连抽冲刺隐藏款→中奖后发布开盒炫耀视频
4. **小美（合成玩家）**：攒了8个重复普通款→使用合成系统→3个R合成1个SR→再用5个SR合成SSR→合成出隐藏款！发帖庆祝
5. **阿杰（裂变达人）**：邀请3个好友助力→解锁免费抽→抽到稀有→分享到群里→好友点击进入→持续裂变循环

### 2.2 头部竞品分析

#### 2.2.1 泡泡玛特（POP MART）——行业标杆

**核心数据**：注册会员超3000万，线下门店400+，机器人商店2000+，年营收超60亿元。

**商业模式**：

| 维度 | 说明 |
|------|------|
| IP策略 | **自有IP矩阵**（Molly、Dimoo、Skullpanda、The Monsters）为核心壁垒 |
| 定价梯度 | 常规59~79元 → 联名79~99元 → 特殊材质99~129元 → MEGA系列799~2999元 |
| 隐藏款概率 | 约1/144（0.69%~1%），部分系列有"大隐藏""小隐藏"之分 |
| 数字化 | 泡泡玛特抽盒机App/小程序：在线抽盒、提示卡、透卡、摇盒模拟、换娃社区 |
| 变现结构 | 盲盒销售占75%，其余为手办、衍生品、BJD、游戏、乐园门票 |
| 二手生态 | 第三方溢价：热门隐藏款5~20倍，雷款低于原价抛售，形成"闲鱼潮玩二级市场" |

**借鉴要点**：
- ✅ 自有IP矩阵 → **我们初期以联名+快速迭代系列为主，中期孵化自有IP**
- ✅ 提示卡/透卡/摇盒 → **数字化付费道具可直接引入**
- ✅ 多价格梯度 → **设计"平民线+精品线+限定线"三档**
- ⚠️ 线下成本重 → **我们纯线上，轻资产运行**

#### 2.2.2 名创优品盲盒——性价比路线

**核心策略**：依赖**外部IP授权**（三丽鸥、迪士尼、漫威、王者荣耀），8000+门店铺货。

| 维度 | 说明 |
|------|------|
| 定价 | 29.9~49.9元（远低于泡泡玛特），吸引学生群体 |
| 隐藏概率 | 1/120~1/144，与泡泡玛特相当但单价低 |
| 数字化 | 基础在线抽盒，无透卡/摇盒等高级功能，无独立二手交易 |
| 劣势 | IP非自有，毛利低；无用户社区粘性；二手溢价微薄 |

**借鉴要点**：
- ✅ 低价策略可以快速拉新 → **设9.9元入门系列做流量入口**
- ⚠️ 纯授权模式不可持续 → **必须逐步建立自有IP**

#### 2.2.3 TOP TOY——潮玩集合店

**定位**：全品类潮玩集合店（盲盒+积木+手办+高达），自有IP（Twinkle、Tammy）+代销。

**数字化特色**：
- 小程序"透卡"：免费查看一个盒的1/3款式信息
- 积分兑换透卡/提示卡
- "潮玩集市"对接闲鱼交易

**借鉴要点**：
- ✅ 透卡机制 → **可作为积分消耗出口和付费道具**
- ✅ 集合店模式 → **引入第三方盲盒品牌抽奖（代理模式）**

#### 2.2.4 手游Gacha系统——概率与商业化精髓

| 品类 | 代表 | 核心机制 |
|------|------|----------|
| 日式扭蛋 | 万代、Taito | 200~500日元/次，无保底，成套收集，整盒优惠 |
| 手游Gacha | 原神、星穹铁道 | **大小保底**、角色UP、概率递增、碎片合成 |
| 社交Gacha | 拼多多、微信小游戏 | 分享抽、助力抽、组队抽、送好友 |

**手游Gacha借鉴核心**：

| 机制 | 说明 | 我们是否采用 |
|------|------|------------|
| **大小保底** | 小保底50%概率不歪，歪后大保底100%出当期UP | ✅ 可引入限定盲盒UP池 |
| **概率递增（软保底）** | 超过阈值后每次+概率 | ✅ 已实现 |
| **硬保底** | 固定次数必出 | ✅ 已实现 |
| **继承机制** | 保底计数跨卡池保留 | ✅ 需实现 |
| **十连打折** | 单抽9折~9.5折 | ✅ 已设计 |
| **体力系统** | 每日免费抽+付费补体力 | ⚠️ 可做每日免费+次数限制 |
| **月卡/战令** | 30元月卡→每日领钻；68元战令→做任务升级 | ✅ 核心商业设计 |
| **首充双倍** | 首次充值双倍货币 | ✅ 标准操作 |
| **碎片/命座** | 重复角色自动转化为碎片溢出 | ✅ 合成系统的基础 |

### 2.3 行业趋势

| 趋势 | 描述 | 对我们影响 |
|------|------|----------|
| **IP为中心** | 泡泡玛特200亿市值核心驱动力是IP | 必须布局自有IP，至少1个原创系列 |
| **监管趋严** | 多地要求概率公示、限制未成年人消费 | 合规必须从第一天做起 |
| **线下线上融合** | 机器人商店+抽盒机+直播拆盒 | 我们纯线上，可做直播开盒 |
| **二手交易生态** | 高价隐藏刺激消费，雷款导致流失 | 交换市场+合成系统解决雷款 |
| **社交裂变升级** | 从简单分享到助力/组队/送礼 | 核心获客手段，必须深度嵌入 |
| **数字藏品降温** | NFT盲盒政策风险高 | 暂不涉及，保持观望 |

### 2.4 我们的差异定位

| 维度 | 泡泡玛特 | 名创优品 | **BOX·MAGIC** |
|------|----------|----------|--------------|
| 形态 | 线下+线上 | 线下+线上 | **纯线上，超轻量** |
| 价格 | 59~79元 | 29.9~49.9元 | **3.9~19.9元（极致性价比）** |
| IP | 自有IP壁垒 | 外部授权 | **自创IP+月度快速迭代** |
| 玩法 | 抽盒+换娃 | 抽盒 | **抽盒+合成+碎片+战令+裂变** |
| 获客 | 门店自然流量 | 门店+IP粉 | **社交裂变+极致性价比** |
| 运营 | 成熟会员+社区 | 基础会员 | **战令+月卡+首充礼包组合拳** |

## 4. 用户端功能

### 4.1 盲盒系列展示

#### 4.1.1 系列列表页

| 功能 | 描述 |

| 系列卡片 | 封面图、系列名称、价格、已售数量、热度标签（🔥 HOT/NEW/限定/即将售罄） |

#### 4.2.3 数字化付费道具

借鉴泡泡玛特和 TOP TOY 的经验，引入以下付费/积分道具提升ARPU：

| 道具名称 | 售价 | 效果 | 设计目的 |
|----------|------|------|----------|

### 4.4 盲盒交换市场

#### 4.4.1 核心匹配逻辑

```
用户A:  我想要 [款式X]  →  用户B: 我有 [款式X] 且想要 [款式A]
        我有多余 [款式A]  ←  用户B: 我多余 [款式X]

匹配成功条件：
  1. A 有 B 想要的，B 有 A 想要的（双向匹配）
  2. 双方均为重复款（拥有数量 ≥ 2 或专门选择用于交换）
  3. 双方稀有度对等（稀有↔稀有，普通↔普通，或平台补贴差价）
```

#### 4.4.2 功能列表

| 功能模块 | 描述 |
|----------|------|
| 发布交换需求 | 选择「我要」的款式 + 「我有」的款式（从库存中选择重复款） |
| 匹配推荐 | 系统自动匹配双向需求，按匹配度排序推送，支持通知提醒 |
| 交换广场 | 浏览所有用户发布的交换需求（匿名展示），可按系列/稀有度筛选 |
| 交换确认流程 | 发起交换→对方确认→系统锁定礼品→双方确认→交换完成→库存更新 |
| 交换记录 | 历史交换记录查询（成功/失败/进行中） |

#### 4.4.3 交换规则

| 规则 | 说明 |
|------|------|
| 同系列交换优先 | 相同系列内交换的手续费更低（5% vs 10% 积分消耗） |
| 跨系列交换 | 允许跨系列，但需支付更高手续费 |
| 交换冷静期 | 成功交换后 24 小时内不可对同一款式再次发起交换（防刷） |
| 稀有度映射 | 系统维护稀有度价值映射表，保障交换公平性 |
| **盲盒盲换** 🆕 | 提交一个未知盲盒，系统随机返还一个同级别未知盲盒（0手续费，增添惊喜感） |

### 4.6 合成/升级系统 🆕

#### 4.6.1 核心机制

消耗若干个低稀有度盲盒，合成一个更高稀有度或限定款。

| 合成方式 | 配方 | 结果 |
|----------|------|------|
| **固定合成** | 3个普通(R) → 1个稀有(SR) | 100%成功，必得随机稀有 |
| **高级合成** | 5个稀有(SR) → 1个隐藏(SSR) | 100%成功，必得随机隐藏 |
| **顶配合成** | 3个隐藏(SSR) → 1个限定(UR) | 100%成功，必得随机限定 |
| **概率合成** | 2个普通(R) → 1个稀有(SR) | 50%成功，失败材料消失 |
| **定向合成** | 3个普通(R) + 指定款碎片x5 → 指定稀有 | 100%定向获得 |

#### 4.6.2 合成树/图鉴

```
                   ┌──────────┐
                   │  限定款UR  │
                   │  合成配方  │
                   └────┬─────┘
                        │  3个隐藏
              ┌─────────▼─────────┐
              │    隐藏款 SSR      │
              │  合成配方: 5稀有    │
              └─────────┬─────────┘
                        │  5个稀有
        ┌───────────────▼───────────────┐
        │          稀有款 SR              │
        │  合成配方: 3普通 / 2普通(概率)  │
        └───────────────┬───────────────┘
                        │  3个普通
              ┌─────────▼─────────┐
              │    普通款 R        │
              │  (抽盒基础产出)     │
              └───────────────────┘
```

#### 4.6.3 合成记录与图鉴

- 每次合成记录保存（类似抽奖记录）
- **合成图鉴**：展示已合成的稀有款式，达成特定合成目标解锁成就
- 合成排行榜：合成稀有数量最多的用户排名

#### 4.6.4 运营价值

| 价值 | 说明 |
|------|------|
| **消耗冗余库存** | 大量重复普通款有了"出口"，减少用户挫败感 |
| **提升用户粘性** | 合成需要积累材料，形成中长期目标 |
| **制造稀缺感** | 高等级合成品是身份的象征 |
| **刺激消费** | "还差2个普通就能合成稀有"→继续抽 |

### 4.7 碎片/拼图系统 🆕

#### 4.7.1 核心玩法

### 4.8 预约/抢购系统 🆕

#### 4.8.1 标准流程

1. **预告** → 小程序首页/社群预告限量款上线时间、数量、价格
2. **预约提醒** → 用户点击"预约"，开售前5分钟推送模板消息
3. **资格验证** → 可设置门槛（会员等级、开盒次数、邀请好友数）
4. **抢购** → 开售瞬间：抽签制优先（服务器压力小）或 先到先得+风控
5. **付款** → 抢到后需在15分钟内付款，否则释放库存

#### 4.8.2 防黄牛机制

| 机制 | 说明 |
|------|------|
| **抽签制** | 预约后随机抽取购买资格，降低服务器压力 |
| **时间锁** | 每轮抢购间隔设冷却时间（同一设备/账号） |
| **转赠锁定** | 限量款抢购后需持有7天才可交换/交易 |
| **身份验证** | 高风险订单需手机验证码确认 |

### 6.2 混合概率模型

采用 **动态递增 + 大/小保底 + UP机制** 的混合概率模型，在保障用户体验的同时控制平台出货成本。

#### 6.2.1 模型四要素

| 要素 | 说明 |
|------|------|
| **固定基础概率** | 每次抽奖的基准概率（见 6.1 分级范围） |
| **软保底 (Soft Pity)** | 连续 N 次未出稀有及以上级别时，每次增加稀有概率 |
| **硬保底 (Hard Pity)** | 最多 M 次必出稀有及以上级别；可选配置隐藏款硬保底 |
| **大小保底 (50/50)** 🆕 | 限时UP池中，首次出稀有有50%概率为当期UP角色；若"歪"了，下次必出UP |

#### 6.2.2 概率递增算法

```
算法描述（伪代码）：

function computeProbabilities(user, series):
    // 1. 获取该系列基础配置
    base = getBaseProb(series)
    rareBase = base.Rare
    
    // 2. 获取用户当前连续未出稀有的次数
    streak = getUserStreak(user, series)
    
    // 3. 检查硬保底
    if streak >= config.HardPityGuarantee:
        // 硬保底触发：必出稀有或以上
        return forceRareOrAbove(series)
    
    // 4. 检查软保底（概率递增）
    pityLevel = max(0, streak - config.SoftPityThreshold)
    rareProb = rareBase + (pityLevel * config.SoftPityIncrement)
    
    // 5. 上限封顶
    rareProb = min(rareProb, config.MaxProbabilityCap)
    
    // 6. 调整普通款概率（稀有概率增加的部分从普通款扣除）
    commonProb = base.Common - (rareProb - rareBase)
    commonProb = max(commonProb, 0.0)
    
    // 7. 隐藏款、限定款保持固定概率（独立计算）
    hiddenProb = base.Hidden
    limitedProb = base.Limited
    
    // 8. 归一化处理
    total = commonProb + rareProb + hiddenProb + limitedProb
    return normalize(commonProb, rareProb, hiddenProb, limitedProb, total)

function computeUP(user, series, drawResult):
    if NOT isUPPool(series):
        return drawResult  // 非UP池，直接返回
    
    if drawResult.rarity == RARE:
        // 抽到稀有级
        if user.hasGuaranteedUP():
            // 大保底激活 → 必出UP款式
            return forceUP(drawResult)
        else:
            // 小保底：50%概率
            if random(0,1) < 0.5:
                return forceUP(drawResult)  // "没歪"
            else:
                user.setGuaranteedUP(true)  // "歪了"，下一次大保底
                return drawResult
    
    return drawResult
```

#### 6.2.3 概率变化示例

| 连续未出稀有次数 | 普通款概率 | 稀有款概率 | 隐藏款概率 | 限定款概率 |
|:---:|:---:|:---:|:---:|:---:|
| 0 | 80.0% | 15.0% | 4.0% | 1.0% |
| 10 | 80.0% | 15.0% | 4.0% | 1.0% |
| 29 | 80.0% | 15.0% | 4.0% | 1.0% |
| **30** (软保底触发) | **78.0%** | **17.0%** | 4.0% | 1.0% |
| 40 | 58.0% | **35.0%** | 4.0% | 1.0% |
| **50** (到达上限) | **58.0%** | **37.0%** (封顶) | 4.0% | 1.0% |
| **60** (硬保底) | — | **100%** (必出稀有+) | — | — |

> **配置参数示例**：SoftPityThreshold=30, SoftPityIncrement=2%, HardPityGuarantee=60, MaxProbabilityCap=37%

## 附录

### A. 技术栈选型理由

| 组件 | 选择 | 理由 |
|------|------|------|
| 语言 | Go 1.24 | 高并发、编译快、部署简单 |
| HTTP | net/http (无框架) | 轻量、无依赖、适合中小项目 |
| 数据库 | MySQL | 关系型数据，事务支持完善 |
| 缓存 | Redis | 分布式锁、计数器、缓存热点数据 |
| 开发存储 | 内存 Map | 快速原型、单元测试友好 |

### B. 抽奖引擎性能指标

| 指标 | 目标值 |
|------|--------|
| 单次抽奖响应时间 | ≤ 100ms (P99) |
| 十连抽响应时间 | ≤ 300ms (P99) |
| 并发抽奖 QPS | ≥ 5000 |
| 库存扣减一致性 | 强一致 (MySQL事务 + Redis锁) |

### C. 商业模式概览 🆕

| 收入来源 | 占比（预估） | 说明 |
|----------|------------|------|

| 月卡/季卡 | 20% | 稳定续费收入 |
| 战令通行证 | 15% | 周期性爆发收入 |
